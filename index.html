Project G.E.P.P.E.T.T.O.

Godot Engine Procedural Production & Evolutionary Text-to-Object

Godot 4GDExtension C++GDScriptAI ResearchCompute ShadersLLM Integration

The Problem

Traditional game asset pipelines are slow, expensive, and break flow. A designer describes what they need, hands it to an artist, waits days, iterates, and repeats. What if Godot could just understand you? That question is what G.E.P.P.E.T.T.O. set out to answer.

The goal was to build a fully automated, closed-loop pipeline — directly inside Godot 4 — that translates a natural language prompt like "Ancient low-poly monolith, moss-covered, physics enabled" into a game-ready, physics-assembled 3D scene node in under 5 seconds.

System Architecture

The pipeline operates across three tightly coupled layers, each responsible for a distinct phase of the asset lifecycle:

[ Natural Language Prompt ] / [ Player Telemetry ] │ ▼ ┌─────────────────────────────────────┐ │ LAYER 1 · SYNTHESIS │ │ Localized LLM → JSON Schema │ │ Latent Diffusion → Vertex Arrays │ └──────────────────┬──────────────────┘ │ Raw Vertex Data ▼ ┌─────────────────────────────────────┐ │ LAYER 2 · ENGINE INTEGRATION │ │ GDExtension C++ · ArrayMesh │ │ RigidBody3D · CollisionShape3D │ └──────────────────┬──────────────────┘ │ Unoptimized Scene Tree ▼ ┌─────────────────────────────────────┐ │ LAYER 3 · OPTIMIZATION LOOP │ │ Edge-Collapse Decimation │ │ Compute Shader Texture Baking │ │ FPS / Draw Call Fitness Check │ └──────────────────┬──────────────────┘ │ ▼ [ Game-Ready Optimized Asset ]

How It Was Built — Stage by Stage

1

Prompt Parsing & Geometry Generation

A local, low-latency LLM parses the design token into a strictly typed JSON schema — detailing mesh characteristics, collision requirements, and physics properties. A latent diffusion model then converts those structural parameters into raw point clouds and unindexed vertex arrays.

2

GDExtension Scene Assembly

Using Godot's C++ GDExtension interface, raw vertex data is injected directly into the engine's memory. SurfaceTool and ArrayMesh index vertices, compute normals, and generate tangents. Based on the JSON payload, the engine dynamically assembles the node hierarchy — if physics is requested, it instantiates a RigidBody3D and auto-generates a convex CollisionShape3D.

3

Self-Optimizing Feedback Loop

Every generated asset must pass a fitness check before entering the scene. An automated edge-collapse decimation algorithm runs via Compute Shader if the vertex budget is exceeded. High-poly details are baked into normal maps via a dynamically compiled VisualShader. Assets that fail FPS or draw call targets are discarded or re-queued.

Core Pipeline Script (GDScript)

@tool extends Node3D class_name AIPipelineManager const PIPELINE_URL = "http://127.0.0.1:8080/v1/synthesize" @export_placeholder("e.g., Ancient low-poly monolith, moss-covered") var asset_prompt: String @export_tool_button("Generate Asset Pipeline") var generate_button = Callable(self, "_on_generate_pressed") func _on_generate_pressed() -> void: if asset_prompt.is_empty(): push_warning("Asset prompt cannot be empty.") return _request_asset_payload(asset_prompt) func _assemble_scene_tree(data: Dictionary) -> void: var root_node: Node3D if data.get("physics_enabled", false): root_node = RigidBody3D.new() else: root_node = Node3D.new() root_node.name = data.get("asset_name", "GeneratedAsset") var mesh_instance = MeshInstance3D.new() mesh_instance.mesh = _build_array_mesh(data.get("vertex_data", {})) root_node.add_child(mesh_instance) if root_node is RigidBody3D: var col = CollisionShape3D.new() col.shape = mesh_instance.mesh.create_convex_shape() root_node.add_child(col) get_tree().edited_scene_root.add_child(root_node) root_node.owner = get_tree().edited_scene_root

AI Code Evaluation Framework

Automated, scalable benchmarking for AI-generated code quality

PythonDockergVisorRabbitMQStatic AnalysisCI/CD

The Problem

As organisations integrate LLMs into software workflows, a dangerous gap emerges: no one is systematically verifying the code these models produce. A function that passes a surface-level glance can hide SQL injection vulnerabilities, infinite loops, or silent logic failures.

This framework was built to close that gap — providing a standardised, automated pipeline to evaluate AI-generated code across three pillars: functional correctness, security vulnerabilities, and runtime performance.

Evaluation Pipeline

[ Incoming Code + Test Cases ] │ ▼ ┌─────────────────────────────────────┐ │ STAGE 1 · STATIC ANALYSIS │ │ AST Parsing · Linting · SAST │ │ Malicious pattern detection │ └──────────────────┬──────────────────┘ │ ▼ ┌─────────────────────────────────────┐ │ STAGE 2 · SANDBOX PROVISIONING │ │ Docker + gVisor isolation │ │ Network air-gapped · 256MB RAM cap │ │ 5s wall-clock execution limit │ └──────────────────┬──────────────────┘ │ ▼ ┌─────────────────────────────────────┐ │ STAGE 3 · DYNAMIC EXECUTION │ │ Unit tests · Property-based tests │ │ stdout/stderr capture │ └──────────────────┬──────────────────┘ │ ▼ ┌─────────────────────────────────────┐ │ STAGE 4 · METRIC AGGREGATION │ │ Pass@k · OWASP scoring │ │ Execution time · Memory profiling │ └──────────────────┬──────────────────┘ │ ▼ [ Structured JSON Report ]

How It Was Built

1

Security-First Sandbox Design

Executing untrusted AI code is inherently dangerous. The sandbox uses gVisor to intercept host system calls, creating a kernel-level security boundary. Every container is network air-gapped, memory-capped at 256MB, and destroyed immediately after execution — no state leakage possible.

2

Multi-Dimensional Scoring

Rather than a simple pass/fail, every piece of code receives scores across three axes: Functional (did it produce correct output?), Security (does it contain OWASP Top 10 vulnerabilities?), and Performance (what's its execution time and memory footprint?). The Pass@k metric evaluates model capability across multiple samples per prompt.

3

Structured JSON Reporting

Every evaluation produces a machine-readable JSON report — designed to plug directly into CI/CD pipelines, dashboards, or LLM fine-tuning workflows. The report includes full test breakdowns, vulnerability details, and performance grades.

Academic Performance Analytics

Predictive BI platform for early student intervention at scale

PythonApache AirflowApache SparkXGBoostFastAPIPostgreSQLReact

The Problem

Student data sits locked in silos — grades in one system, attendance in another, LMS engagement in a third. By the time an institution recognises a student is struggling, it's already too late to meaningfully intervene. Dropout rarely happens suddenly. It happens gradually, invisibly, across disconnected systems.

This system was built to connect those dots — aggregating all academic signals into a single predictive engine that flags at-risk students weeks before failure, giving advisors the window they need to act.

System Architecture

[ SIS · LMS · Attendance Logs ] │ (REST APIs / CSV ingestion) ▼ ┌─────────────────────────────────────┐ │ ETL PIPELINE │ │ Apache Airflow (orchestration) │ │ Apache Spark (transformation) │ │ Data validation & PII masking │ └──────────────────┬──────────────────┘ │ ▼ ┌─────────────────────────────────────┐ │ DATA WAREHOUSE │ │ PostgreSQL · Star schema │ │ dim_students · fact_performance │ └──────────────────┬──────────────────┘ │ ▼ ┌─────────────────────────────────────┐ │ ML ENGINE (Python · XGBoost) │ │ Risk Score 0–100 per student │ │ Feature: attendance, GPA, LMS freq │ │ Recall target ≥ 0.85 │ └──────────────────┬──────────────────┘ │ ▼ ┌─────────────────────────────────────┐ │ DASHBOARD · FastAPI + React/D3.js │ │ Role-based: Admin · Teacher · Student│ │ Auto-alerts when Risk Score > 75 │ └─────────────────────────────────────┘

How It Was Built

1

ETL Pipeline & Data Integration

Built automated ingestion pipelines using Apache Airflow for scheduling and Apache Spark for distributed transformation. Data flowed in from Student Information Systems and Learning Management Systems via REST APIs and flat-file CSVs, with strict validation rules flagging corrupt or missing records before processing.

2

Predictive Risk Model

An XGBoost classifier was trained on five engineered features — attendance rate, assignment submission delay, LMS login frequency, midterm scores, and historical GPA. The model outputs a Risk Score from 0–100, predicting failure probability at the course midpoint. Achieved a Recall score of 0.87, minimising false negatives for at-risk students.

3

Role-Based Dashboard & Alert System

Built a React/D3.js dashboard with three access tiers: Administrators see institutional KPIs, Educators see per-course risk alerts and intervention logs, Students see their own performance trajectory and a "what-if" grade calculator. Automated email alerts fire when any student crosses Risk Score 75.

Unity Gameplay Prototypes

Physics-based mechanics from concept to playable greybox — fast

UnityC#Rigidbody PhysicsScriptableObjectsNew Input SystemProBuilder

The Philosophy

The most expensive mistake in game development is building beautiful things that aren't fun. Every prototype in this series was built around a single principle: validate the feel before committing to the art.

Each project took a single mechanical idea — a grappling hook, a physics-based puzzle, a movement system — and built it into a fully playable, testable greybox prototype using a strict architecture designed for speed, modularity, and zero debt.

Architecture Pattern

┌──────────────────┐ │ Player Input │ (Unity New Input System) └────────┬─────────┘ │ Input Actions ▼ ┌──────────────────────┐ │ PlayerController │ (Physics Logic Layer) └────────┬─────────┬───┘ │ │ ▼ ▼ ┌──────────────┐ ┌──────────────────────┐ │ PlayerData │ │ Feedback/FX Manager │ │ (ScriptableObj)│ │ PlayScreenShake() │ │ moveSpeed │ │ PlayParticles() │ │ jumpForce │ │ PlaySFX() │ │ gravityMult │ └──────────────────────┘ └──────────────┘ (Tweakable in editor — no recompile)

Separating Data (ScriptableObjects), Logic (MonoBehaviours), and Feedback (FX Manager) means designers can tune feel in real-time without touching code, and the same movement script can drive both a player and an AI enemy without modification.

How It Was Built

1

Physics-Based Movement (Not CharacterController)

Most tutorials use Unity's CharacterController — which feels floaty and ignores physics. These prototypes use a Rigidbody with clamped velocity changes via ForceMode.VelocityChange, giving natural momentum, mass interaction, and friction. Custom gravity multipliers make jumping feel weighted and responsive — falling faster than rising, with short-hop support.

2

ScriptableObject Data Layer

Player movement parameters — speed, jump force, gravity multipliers — live in a ScriptableObject asset, not hardcoded values. This means a designer can open the Inspector, drag a slider, and feel the change in real time during a play session, without stopping, modifying code, and recompiling.

3

Greybox Environment & Playtest Metrics

Environments were built in ProBuilder on a strict 1m grid snap using the standard greybox colour convention: white for traversable floors, charcoal for walls, orange for hazards, neon green for interactables. Each prototype was playtested against four measurable criteria before advancing to production.

Core Movement Controller (C#)

[RequireComponent(typeof(Rigidbody))] public class PrototypePlayerController : MonoBehaviour { [SerializeField] private PlayerMoveConfig config; private Rigidbody rb; private Vector2 inputDirection; private bool isGrounded; void FixedUpdate() { MovePlayer(); ApplyCustomGravity(); } private void MovePlayer() { Vector3 targetVelocity = new Vector3( inputDirection.x * config.moveSpeed, rb.velocity.y, inputDirection.y * config.moveSpeed); Vector3 velocityChange = targetVelocity - rb.velocity; velocityChange.x = Mathf.Clamp(velocityChange.x, -config.maxAcceleration, config.maxAcceleration); velocityChange.z = Mathf.Clamp(velocityChange.z, -config.maxAcceleration, config.maxAcceleration); velocityChange.y = 0; rb.AddForce(velocityChange, ForceMode.VelocityChange); } private void ApplyCustomGravity() { // Snappy fall: heavier on the way down if (rb.velocity.y < 0) rb.velocity += Vector3.up * Physics.gravity.y * (config.downwardGravityMultiplier - 1) * Time.fixedDeltaTime; } }
